# 길눈 실제형 의미 안내 데모 v2 설계

## 목표

현재의 “큰 정답 버튼을 누르는 3단계 튜토리얼”을 실제 공공서비스처럼 메뉴와
선택지가 많은 합성 민원 화면으로 바꾼다. 사용자는 먼저 스스로 진행하고, 막혔을
때 길눈의 도움을 요청한다. 길눈은 버튼 좌표가 아니라
`service + revision + checkpoint + stableKey + role + accessibleName + expectedState`
계약으로 현재 해야 할 대상을 다시 찾는다.

앱 홈과 연습 화면에서 다음 두 문장을 핵심 가치로 함께 노출한다.

> 화면이 바뀌어도, 해야 할 일을 다시 찾아드려요.
>
> 좌표는 빗나가도, 의미는 다시 찾습니다.

## 확인된 문제

- 각 단계에서 실제로 누를 수 있는 진행 버튼이 하나뿐이라 정답이 처음부터 보인다.
- 나머지 항목은 대부분 읽기 전용이어서 사용자가 복잡한 화면을 해결했다는 감각이 없다.
- A/B 배치 변경이 수동 테스트 버튼으로 노출되어 실제 사이트 업데이트처럼 느껴지지 않는다.
- 의미 기반 복구가 도움을 받은 뒤의 강조 표시로만 나타나 좌표 기반 안내와의 차이가 약하다.
- 정부24형, 복지로형, 건강보험형의 색은 다르지만 폼 구조는 거의 동일하다.

## 검토한 접근과 결정

### 1. 정적인 정부24 화면 복제

정부24와 비슷한 색, 탭, 섹션을 한 화면에 그리는 방식이다. 외형은 실제처럼 보이지만
레이아웃 변화 내성과 의미 기반 안내의 장점은 보여주지 못한다.

### 2. 사용자가 누르는 A/B 배치 전환

현재처럼 “화면 배치 바꿔보기” 버튼으로 두 레이아웃을 비교한다. 테스트에는 유용하지만
사용자가 일부러 만든 데모라고 느끼기 쉽다. 릴리스의 주 흐름에서는 제거하고 테스트
빌드의 보조 기능으로만 유지한다.

### 3. 진행 중 사이트 업데이트와 의미 복구 — 선택

첫 체크포인트를 마친 직후 사이트가 업데이트된 것처럼 A에서 B로 자동 재배치한다.
버튼 위치, 섹션 순서, 주변 항목은 바뀌지만 의미 계약은 유지된다. 이후 도움을 요청하면
길눈이 새 위치의 정확한 대상 하나를 찾아 강조한다. 사용자가 승인한 최종 방식이다.

## 실제형 화면 원칙

- 각 서비스는 서로 다른 기관형 정보 구조를 사용한다.
- 현재 체크포인트에는 진행 대상 외에도 4개 이상의 그럴듯한 메뉴나 선택지를 둔다.
- 라디오, 셀렉트, 탭, 아코디언, 보조 버튼은 실제로 상태가 변한다. 단순 장식이나
  비활성 카드로 정답을 노출하지 않는다.
- 잘못된 선택은 실제 처리 없이 합성 폼 상태만 바꾸며, 오류 대신 “이 연습의 다음
  순서는 아니에요”처럼 중립적으로 안내한다.
- 자동 안내는 사용자가 요청하기 전에는 나타나지 않으며, 대상을 대신 클릭하지 않는다.
- 개인정보 입력칸, 실제 인증, 실제 자격 판정, 기관 전송, 결제, 파일 발급은 만들지 않는다.
- 모든 사용자·주소·문서·조회 결과에는 `(가상)` 또는 `(모의)`를 표시한다.

## 화면 구성

### 공통 네이티브 영역

- 홈의 상단에는 제공된 길눈 로고와 핵심 가치 두 문장을 표시한다.
- 서비스 카드는 “복잡 화면 연습”이라는 표현 대신 실제 과업과 어려운 지점을 설명한다.
  예: `주민등록표 등본 · 문서 종류와 수령 방법 찾기`.
- 연습 화면의 하단 도크는 `안내 읽기`와 `길눈에게 찾아달라고 하기` 두 동작으로 구성한다.
- 도움을 받은 후에는 `예전 위치가 아니라, 이름과 역할을 확인해 찾았어요`라는 영수증을
  한 번만 표시한다.
- `연습용 화면 · 실제 기관과 연결되지 않아요` 배너는 네이티브 도크 위와 WebView
  상단에서 항상 보인다.

### 정부24형 주민등록표 등본

공개된 정부24의 상단 유틸리티, 민원 메뉴, 문서 탭, 안내 패널, 주소·발급·수령 섹션의
정보 밀도를 참고한다. 공식 로고, 원본 에셋, 실제 로그인 화면과 실제 기관 명의는
복제하지 않는다.

진행 체크포인트는 다음 다섯 개다.

1. 네 개의 문서 탭에서 `주민등록표 등본` 선택
2. 주소 섹션에서 `서울특별시(가상)`와 `길눈구(가상)` 선택 후 `주소 확인`
3. 발급 형태에서 `발급(모의)` 선택
4. 수령 방법 목록에서 `온라인발급(본인출력·연습용)` 선택
5. 접힌 안내와 취소·이전·임시저장 사이에서 `민원 신청 연습 마치기` 선택

한 화면에는 다음 요소가 함께 보인다.

- 주민등록표 등본·초본·영문 등본·영문 초본 탭
- `알려드립니다` 아코디언
- 시·군구 두 개의 선택 상자
- 발급·선택발급 라디오
- 온라인발급·전자문서지갑·등기우편·방문수령 선택 상자
- 이전, 임시 저장, 취소, 민원 신청 연습 마치기 버튼
- `법적 효력 없음` 워터마크가 있는 모의 문서 요약

완료 문구는 `발급 연습을 마쳤어요`이며 실제 문서가 만들어졌다는 표현은 사용하지 않는다.

### 복지로형 기초연금

복지서비스 경로, 신청인 요약, 신청 관계, 연락 방법, 확인 항목, 임시 저장을 함께
배치한다.

1. 서비스 목록에서 `기초연금 신청 연습` 선택
2. 신청인 카드에서 `연습 사용자(가상)` 확인
3. 신청 관계에서 `본인이 신청해요` 선택
4. 합성 연락 방법과 안내 수신 여부 확인
5. 임시 저장·이전 사이에서 `신청 내용 확인` 선택

자격 여부나 예상 금액은 보여주지 않는다. 완료 문구는 `신청 연습을 마쳤어요`다.

### 건강보험형 건강검진

개인민원 사이드 메뉴, 조회 사용자, 기준 연도, 검진 종류, 주의 안내와 조회 버튼을
함께 배치한다.

1. 개인민원 메뉴에서 `건강검진 대상 조회 연습` 선택
2. `연습 사용자(가상)` 확인
3. 연도 목록에서 `2026년(가상)` 선택
4. 검진 종류에서 `일반건강검진(모의)` 선택
5. 일정 안내·이전 사이에서 `건강검진 대상 조회` 선택

결과는 `일반건강검진 대상(모의)`까지만 표시하고 의료 수치, 예약, 실제 판정은
다루지 않는다. 완료 문구는 `대상 조회 연습을 마쳤어요`다.

## 레이아웃 변화 내성 시연

### 전환 시점

각 서비스의 첫 체크포인트가 성공하면 다음 순서로 전환한다.

1. 900ms 이내의 비차단 알림으로 `사이트 화면이 업데이트되어 배치가 바뀌었어요` 표시
2. WebView가 같은 서비스와 다음 체크포인트를 유지한 채 Layout B로 재배치
3. 섹션 순서, 주요 버튼 위치, 사이드 메뉴 위치와 주변 선택지 순서를 변경
4. `도움이 필요해요`를 누르기 전에는 대상 강조를 표시하지 않음
5. 도움 요청 시 새 DOM에서 의미 계약이 정확히 하나 일치하는 대상만 노란색으로 강조

### 사용자가 보는 설명

- 전환 전: `지금 화면의 위치를 외우지 않아도 괜찮아요.`
- 전환 직후: `화면 위치가 달라졌어요. 필요한 항목을 찾아보세요.`
- 자동 안내 후: `예전 좌표가 아니라 이름·역할·다음 단계를 확인해 다시 찾았어요.`
- 직접 선택과 다음 체크포인트 확인 후:
  `의미로 찾기 → 직접 선택 → 다음 화면 확인`

좌표 실패를 흉내 내기 위해 엉뚱한 버튼을 강조하거나 자동 클릭하지 않는다. 이전 위치에는
작은 비대화형 설명 표식만 잠시 표시할 수 있으며 스크린리더 탐색 대상에서는 제외한다.

### 의미 계약

- Layout A와 B의 시각 위치와 DOM 그룹 순서는 달라진다.
- 현재 진행 대상의 `stableKey`, `role`, `accessibleName`, `expectedState`는 유지한다.
- 같은 `stableKey`를 가진 복제 요소를 만들지 않는다.
- 하나도 일치하지 않거나 둘 이상 일치하면 실패 폐쇄하고 아무것도 강조하지 않는다.
- checkpoint 변경은 실제 사용자 조작과 정확한 다음 checkpoint가 모두 관찰된 경우에만
  검증 영수증을 만든다.

## 안내와 마찰 감지

- 각 체크포인트는 정확히 하나의 기본 `PatchV1` 진행 대상을 갖는다.
- 화면의 선택지는 모두 조작 가능하지만 서비스별 지정된 비진행 동작만 브리지 V2의
  `NON_PROGRESS` 마찰 이벤트로 보낸다.
- 6초 안에 같은 비진행 동작을 세 번 하거나 도움 버튼을 직접 누르면 도움 선택지를 연다.
- 자동 안내는 현재 Layout B의 검증된 대상을 강조만 한다.
- 도우미 인계는 네이티브 화면에서 대상 이름만 확인하고, 어르신이 WebView에서 직접 누른다.
- 거절 후 재제안 대기시간은 30초를 유지한다.

## 모션

- 기존 로고 스플래시와 서비스 로딩 모션을 유지한다.
- Layout A→B 전환은 페이드와 섹션 이동을 250–450ms로 제한한다.
- 모션 중에도 뒤로가기와 하단 도움 도크는 사용할 수 있어야 한다.
- 시스템 애니메이션 비활성화 또는 `prefers-reduced-motion`에서는 즉시 배치만 바꾼다.
- 레이아웃 변경 모션은 의미 계약이나 checkpoint를 변경하지 않는다.

## 안전과 권리 경계

- 정부24 공개 화면은 정보 구조와 상호작용 밀도의 참고 자료로만 사용한다.
- 정부24 로고, 원본 아이콘, 원본 CSS·JavaScript, 인증 화면, 실제 기관 문구를 복사하지 않는다.
- 상단에는 `정부24와 연결되지 않은 길눈 연습 화면`을 명확히 표시한다.
- `INTERNET` 권한, 외부 이동, 실제 도메인 호출, 쿠키, DOM 저장소, 파일 접근,
  `addJavascriptInterface`를 추가하지 않는다.
- 앱 에셋은 계속 CSP `connect-src 'none'; form-action 'none'` 아래에서 오프라인으로 동작한다.
- 20sp/20px 이상 본문, 22sp 굵은 주요 버튼, 56dp/px 조작 영역과 200% 글자 배율
  대응을 유지한다.

## 상태와 버전

- 앱은 항상 홈에서 시작하며 폼 선택과 Layout B 상태는 재실행 후 복원하지 않는다.
- 도움 수준과 검증 영수증만 기존 schema v2 범위 안에서 저장한다.
- 새 APK는 기존 서명키를 유지하고 업데이트 설치가 가능하도록 `0.2.1 (3)`으로 올린다.
- 기존 `0.2.0 (2)`가 설치되어 있어도 사용자 합성 입력값이나 WebView 폼 상태는 이전하지 않는다.

## 검증

### 단위·정적 검증

- 3개 서비스의 15개 진행 체크포인트와 의미 대상 고유성
- 각 체크포인트에 4개 이상의 실제 조작 가능한 주변 선택지가 존재
- 도움 전에는 강조가 없고 도움 후에는 하나만 강조
- 자동 안내가 클릭이나 폼 값 변경을 하지 않음
- 첫 checkpoint 후 A→B 전환, checkpoint 보존과 semantic target 보존
- A/B에서 대상의 픽셀 위치와 그룹 순서가 실제로 달라짐
- 중복 대상, 교차 서비스, 잘못된 revision과 잘못된 next checkpoint 실패 폐쇄
- 실제 기관 URL, 로고, 로그인, 개인정보 입력, 제출, 결제 요소 부재
- CSP `connect-src 'none'`, `form-action 'none'`

### 계측 검증

- 각 서비스 무도움 완주
- 주민등록표 등본의 다섯 선택을 모두 수행
- A→B 자동 전환 뒤 직접 찾기
- A→B 자동 전환 뒤 자동 안내
- 도우미 인계 후 사용자 직접 선택
- Layout B의 위치 변화와 의미 일치
- 오프라인 로딩과 외부 URL 차단

### 릴리스 게이트

- `testDebugUnitTest`
- `lintRelease`
- debug/release/androidTest APK 빌드
- 정적 개인정보·네트워크·브리지·CSP 검사
- zipalign과 apksigner 검증
- 기존 인증서 SHA-256 일치
- 기기가 없으면 TalkBack, TTS, 150/200% 글자, 모션, 업데이트 설치를
  `DEVICE_BLOCKED`로 분리
